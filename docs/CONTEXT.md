## Project

Building a full Intercom/Zendesk-style customer support platform end-to-end:
auth/team mgmt, embeddable live chat widget, email channel, unified inbox,
knowledge base, AI conversation summarization, custom domains — plus
production-grade concerns (real-time, queues, security, multi-tenancy).

Confirmed constraints: no real credentials (stub email provider + AI gracefully),
local-only via Docker Compose, TypeScript stack throughout (Fastify + Postgres/
Prisma + Redis/BullMQ + Socket.IO backend; React/Vite/Tailwind dashboard;
vanilla-TS embeddable widget).

### `packages/widget/` — embeddable loader architecture

**Deviation from the original plan's phrasing** ("widget iframe app (React, separate small bundle)"): rather than a second React app, `packages/widget` contains **only** `src/loader.ts` — a dependency-free vanilla-TS script that builds (via `vite.config.ts`, lib mode, IIFE) straight into `packages/web/public/tele-widget.js`. The actual chat UI is `packages/web/src/pages/WidgetFrame.tsx`, a route in the **same** dashboard Vite app, served at `/widget-frame`. Rationale: reuses the dashboard's existing build tooling, Tailwind setup, and Socket.IO client wiring instead of duplicating it in a second app; the end result is identical from a customer's perspective — one `<script src=".../tele-widget.js" data-workspace="...">` tag, and the loader injects a bubble + iframe pointing back at our own origin. Verified end-to-end with Playwright: embed on a plain HTML page → open bubble → send message → appears in dashboard inbox → **reload the host page → chat history still there** (visitor JWT persisted in the iframe's own localStorage, scoped to `tele.visitorToken.<workspaceSlug>`).

If asked to explain this in review: the "single script tag, any website" requirement is about what the *customer* does, not our internal package boundaries — this fully satisfies it while being less code to maintain.

## Key architectural decisions already made (see plan file for full rationale)

- Route convention: dashboard routes are `/api/workspaces/:workspaceId/...`, gated by `requireWorkspaceMember(role?)`. Public widget routes are `/api/public/widget/...`, gated by `authenticateVisitor` (Bearer visitor JWT, not cookies — widget is 3rd-party embedded).
- `createMessage` in `modules/messages/service.ts` is the **single choke point** for writing a message regardless of source (agent REST reply, widget REST send, future email-inbound processor) — it sanitizes, persists, broadcasts over both `/agent` and `/widget` namespaces to the conversation room, and enqueues the debounced summarize job. Any new message-writing path (e.g. email inbound) should call this function, not insert `Message` rows directly.
- Socket.IO is a live-update fast path only; REST (`GET .../messages?after=`) is the source of truth for catch-up after reconnects. Message ordering is `(createdAt, id)`.
- Snoozed→Open transition on new contact message is handled inside `createMessage` (`status: OPEN` when `senderType === CONTACT`). A separate repeatable job is still needed for snooze **expiry** (`snoozedUntil` passing with no new message) — not yet built, goes in `worker.ts`.
- Frontend data flow: React Query is the client cache for everything; Socket.IO handlers call `queryClient.setQueryData`/`setQueriesData` directly to patch the cache on live events (`Inbox.tsx`'s `conversation:updated` handler, `ConversationDetail.tsx`'s `message:new`/`message:read`/`typing:*`/`conversation:summary_updated` handlers) rather than blindly invalidating+refetching — keeps the UI responsive without extra round-trips.

## Gotchas hit so far (avoid re-debugging these)

1. **npm workspaces + `allowScripts`**: this npm version blocks postinstall scripts by default. Had to run `npm approve-scripts <pkg>` for `@prisma/client`, `@prisma/engines`, `esbuild`, `msgpackr-extract`, `prisma`, `fsevents` (macOS-only, safe). This wrote an `allowScripts` block into root `package.json` — that's expected/needed, don't remove it.
2. **ioredis default import under `moduleResolution: NodeNext`**: `import IORedis from "ioredis"` type-errors ("has no construct signatures"). Fix: use the named export instead — `import { Redis } from "ioredis"`.
3. **BullMQ bundles its own nested `ioredis`** (different version than our top-level one) — passing a live `Redis` client instance we created into `new Queue(name, { connection })` fails to typecheck (deep structural mismatch between the two ioredis copies' internal types). Fix: for BullMQ, pass a **plain options object** (`bullConnectionOptions` in `queues/connection.ts`, parsed from `REDIS_URL` into `{host, port, password, username}`), not a client instance. Reserve the real shared `Redis` client (`redisConnection`) for things that want an actual client to `.duplicate()`, e.g. the Socket.IO Redis adapter.
4. **`@fastify/cookie` type augmentation** (adds `req.cookies`, `reply.setCookie/clearCookie`) only becomes visible to the TS project once some file actually imports `@fastify/cookie` — that happens in `server.ts`. If you typecheck a file in isolation before `server.ts` exists/imports it, you'll see spurious "Property 'cookies' does not exist" errors; not a real bug, just needs the full program compiled together.
5. **`@types/cookie`** had to be added explicitly as a devDependency (the `cookie` package used for parsing Socket.IO handshake cookies in `agentNamespace.ts` doesn't ship its own types at the installed version).
6. **Fastify v5 custom logger**: use `Fastify({ loggerInstance: pinoInstance })`, not `logger:`.
7. **Presence tracking is in-memory** (per-process `Map`s in `realtime/presence.ts`), which only gives correct online/offline status on a single API node. Documented in-code and here as a known scale limitation — production multi-node would move this to Redis (already have the adapter's Redis connections available to reuse). Don't "fix" this later without updating docs/ARCHITECTURE.md's scale section to match.
8. **Host already has native Postgres (5432) and Redis (6379) running** (Homebrew services, not ours) — docker-compose's default ports collided. Remapped in `docker-compose.yml`: Postgres host port **5433**, Redis host port **6380** (container-internal ports unchanged). `packages/api/.env` (the real, gitignored dev file) points at `5433`/`6380`; `.env.example` keeps the vanilla `5432`/`6379` defaults since that port clash is specific to this machine. Maildev's `1025`/`1080` had no conflict and are unchanged.
9. **`z.coerce.boolean()` on `COOKIE_SECURE=false` evaluated to `true`** — `Boolean("false")` is `true` in JS; `z.coerce.boolean()` just calls `Boolean()`. Found this by curling signup and seeing `Secure` in the `Set-Cookie` header despite `.env` saying `false`. Fixed with a `z.enum(["true","false"]).transform(v => v === "true")` helper (`booleanString` in `env.ts`) instead. If you add more boolean env vars, reuse `booleanString`, don't reach for `z.coerce.boolean()`.
10. **CSRF check blocked signup/login themselves** (chicken-and-egg: no `csrf_token` cookie exists until *after* a session is established). Fixed in `lib/csrf.ts`: `verifyCsrf` now also passes through when there's no existing `session` cookie — there's no authenticated state yet for a cross-site request to ride on, and `SameSite=Lax` already stops cross-site cookie-bearing POSTs, which is what actually prevents login/signup CSRF. The double-submit token only matters once a session exists, e.g. for the invite-creation call, which correctly still requires the CSRF header — verified via curl.
11. **Form `<label>`s need `htmlFor`/`id`** — Playwright's `getByLabel()` (and real screen readers) need the label/input association; plain visually-adjacent `<label>` text doesn't count. Hit this writing the first Playwright smoke test against `Signup.tsx`/`Login.tsx`/`AcceptInvite.tsx`/`TeamSettings.tsx` invite form — added `htmlFor`/`id` pairs to all of them. Keep doing this for every new form field.
12. **BullMQ v5 job IDs cannot contain `:`** ("Custom Id cannot contain :") — `enqueueSummarize`/`enqueueOutboundEmail` originally used `summarize:${id}`/`outbound:${id}` as jobId and silently failed to enqueue (caught by our own try/catch + `logger.warn`, so it didn't crash anything, just meant summarize/outbound-email jobs never actually ran — found via the worker log, not a thrown exception at the call site). Fixed by switching to `-` as the separator. If you add more custom jobIds, use `-` or no separator at all, never `:`.
13. **`plainto_tsquery` ANDs every remaining word together — too strict for natural-language "suggest as you type."** Typing "how do I **get** a refund" matched zero articles against one titled "How do I **request** a refund?" because `plainto_tsquery('english', 'how do i get a refund')` produces `'get' & 'refund'` (English stopwords like "how"/"do"/"i"/"a" are dropped, but "get" is NOT a stopword and isn't in the document at all, so the AND fails even though "refund" matches great). Found via a Playwright test that filled the widget composer and waited for the suggestions box — it never appeared; curl'ing the search endpoint directly with the same query returned `{"articles":[]}`, confirming it wasn't a frontend bug. Fixed in `modules/kb/service.ts`'s `searchPublicArticles`: use `plainto_tsquery` only to safely tokenize/stem/quote the raw user string (it's the injection-safe entry point), then flip its `&` operators to `|` via a string `replace` on **Postgres's own output** (never on the raw user string, so this can't be turned into a query-syntax injection) before casting back to `tsquery`. This makes both the public KB search and the widget suggester match on *any* significant word, ranked by `ts_rank` — the right behavior for a help-center suggester, where "some overlap" should surface a result, not require an exact AND match. If you touch this query again, keep the OR semantics; don't "simplify" it back to `plainto_tsquery` directly.
15. **`@anthropic-ai/sdk@0.32.1` (the version originally scaffolded) predates structured outputs** (`messages.parse`, `zodOutputFormat`, the `helpers/` directory didn't exist). Bumped to `^0.110.0`. If you see "no exported member `parse`" or a missing `helpers/zod` module, check the installed version — this is a fast-moving SDK and pinning an old minor will silently lose recent features rather than error at install time.
16. **`zodOutputFormat` types against `zod/v4`, not the v3 API the rest of this app uses.** Zod 3.23+ ships a `zod/v4` subpath (a preview of the v4 API within the v3 package) specifically for this kind of gradual adoption — `modules/ai/summarize.ts` imports `{ z } from "zod/v4"` *only in that file* to build the schema passed to `zodOutputFormat`; everywhere else in the app keeps using plain `"zod"` (v3). Don't "fix" this by switching the whole project to `zod/v4` — it's an intentionally scoped import.
17. **No Anthropic API key available in this environment** — the real Claude call path in `modules/ai/summarize.ts` is written directly from the current `claude-api` skill's TypeScript reference (model ID, `messages.parse`, `zodOutputFormat`, per-request `timeout` in ms) but has only been verified via the **no-key stub branch** (logs and returns early — confirmed live: enqueue → debounce → worker picks it up → logs `"ANTHROPIC_API_KEY not set — skipping AI summarization (stub mode)"` → no crash). If a key becomes available, the first real call is the thing to check — specifically that `response.parsed_output` is populated and matches `ConversationSummaryPayload`.
18. **Playwright's `text=` selector matches `<input>`/`<textarea>` *value*, not just rendered text content.** Cost real debugging time: a test did `click("Send")` then immediately `waitForSelector("text=<the message>")` and it resolved instantly — but it had matched the *composer textarea's own stale value*, not the sent message bubble, because the async send hadn't completed yet. The message send was actually fine; the test was a false positive that looked like a bug (composer not clearing, message missing from thread). Lesson: after a click that triggers an async network call, wait on something that can *only* exist post-response (e.g. `page.waitForTimeout` plus checking `composer.inputValue() === ""`, or waiting on a selector scoped away from any input/textarea), not a bare `text=` match if a form field with similar content is still on screen.
